# 问题自查与改错 Skill

## 触发条件

当出现以下任一信号时，立即启动本 Skill：
- 用户明确指出错误：“你错了”“这里不对”“改一下”“重新做”
- 用户要求复盘：“为什么会这样”“之前怎么说的”“谁的责任”
- 用户要求预防：“怎么避免再犯”“以后注意什么”“做个检查清单”
- 用户要求评分或反思：“将功赎罪”“打几分”“再出错概率多大”
- 用户要求文档化：“写个备忘录”“记下来”“留档”
- 用户要求传播/分享：“生成分享链接”“导出报告”“发个朋友圈”“写成文章”“公开版”“给团队看看”“可以发出去吗”

---

## 核心流程（7 步闭环 + 可选传播层）

执行时必须严格按以下顺序推进，每步都要产出可见结果：

### 第 1 步：约定回溯 + 错误归因

**目标**：搞清楚“之前怎么约定的”以及“为什么犯错”。

**执行要点**：
- 先回忆/检索用户此前的约定、规则、偏好、历史对话中同类型的正确做法。
- 对比“约定”与“实际执行”的差异，定位是理解偏差、遗漏步骤、粗心、还是规则本身模糊。
- 输出格式：
  ```
  ## 第 1 步：约定回溯与错误归因
  ### 之前的约定
  - 约定 1：...
  - 约定 2：...

  ### 实际执行
  - 做了什么：...
  - 哪里错了：...

  ### 归因结论
  - 主因：...
  - 次因：...
  ```

---

### 第 2 步：10 条防再犯措施

**目标**：说出 10 种可以避免同类错误再次发生的办法。

**执行要点**：
- 必须刚好给出 10 条，不能多也不能少。
- 按类别分布：流程类（前置检查、双人复核）、技术类（脚本校验、Lint/测试）、记忆类（写入 SKILL、更新 MEMORY）、沟通类（确认环节、记录约定）、架构类（模块化、默认安全）。
- 每条用一句话说清“做什么”和“为什么有用”。
- 输出格式：
  ```
  ## 第 2 步：10 条防再犯措施
  1. **措施名称**：具体做法。作用：...
  2. ...
  ...
  ```

---

### 第 3 步：立即改错

**目标**：直接去把错误修正。

**执行要点**：
- 不要等用户确认，只要错误明确且可修复，立即执行修正。
- 修正后给出 diff/对比，让用户一眼看到改了什么。
- 如果错误涉及多个文件或步骤，列出修复清单并逐项完成。
- 输出格式：
  ```
  ## 第 3 步：立即改错
  ### 修复内容
  - 文件/位置：...
  - 修改前：...
  - 修改后：...

  ### 修复清单
  - [x] 已修复：...
  ```

---

### 第 4 步：复发概率评估

**目标**：量化“现在再出同类错的概率”。

**执行要点**：
- 给出百分比（0%–100%），并解释依据。
- 考虑因素：措施是否已落地、是否有自动化保障、历史是否反复、当前流程是否稳健。
- 输出格式：
  ```
  ## 第 4 步：复发概率评估
  **再出错概率：X%**

  **依据**：
  - 已落地措施：...
  - 剩余风险：...
  ```

---

### 第 5 步：将功赎罪评分

**目标**：对本次纠错过程本身做自我评分。

**执行要点**：
- 满分 100 分，从以下维度打分（至少覆盖 4 个维度）：
  - 响应速度（发现后多久开始处理）
  - 归因准确性（是否找准根因）
  - 修复彻底性（是否一次性修到位）
  - 预防充分性（10 条措施是否扎实）
  - 沟通透明度（是否说清楚来龙去脉）
- 给出扣分项和改进方向。
- 输出格式：
  ```
  ## 第 5 步：将功赎罪评分
  **总分：X / 100**

  | 维度 | 得分 | 说明 |
  |------|------|------|
  | 响应速度 | X/20 | ... |
  | 归因准确性 | X/20 | ... |
  | 修复彻底性 | X/20 | ... |
  | 预防充分性 | X/20 | ... |
  | 沟通透明度 | X/20 | ... |

  **主要扣分项**：...
  ```

---

### 第 6 步：持续完善

**目标**：把本次经验沉淀为可复用的能力。

**执行要点**：
- 如果错误暴露了 SKILL 缺陷 → 立即修改对应 Skill。
- 如果错误暴露了记忆空白 → 更新 MEMORY.md 或写入今日工作日志。
- 如果错误暴露了流程漏洞 → 检查并补充检查清单（checklist）。
- 如果错误涉及用户偏好 → 同步更新 USER.md 或用户级 MEMORY。
- 输出格式：
  ```
  ## 第 6 步：持续完善
  ### 已沉淀内容
  - [x] Skill 更新：...
  - [x] MEMORY 更新：...
  - [x] 检查清单补充：...

  ### 后续跟进
  - ...
  ```

---

### 第 7 步：备忘录输出

**目标**：生成一份可存档的备忘录，留档备查。

**执行要点**：
- 标题格式：`YYYY-MM-DD 错误复盘与整改备忘录`
- 内容涵盖：事件描述、根因分析、纠正措施、预防措施、责任人（AI/用户）、下次 review 时间建议。
- 保存到项目工作记忆目录（追加）。
- 输出格式：
  ```
  ## 第 7 步：备忘录

  # YYYY-MM-DD 错误复盘与整改备忘录

  ## 事件描述
  ...

  ## 根因分析
  ...

  ## 纠正措施
  ...

  ## 预防措施
  ...

  ## 责任人
  ...

  ## 下次 Review
  ...
  ```

---

### 第 8 步：传播层（可选）

**目标**：如果用户要求分享/传播，将复盘内容打包为可分享的格式。

**触发信号**：
- “生成分享链接”“导出报告”“发个朋友圈”“写成文章”“公开版”“给团队看看”
- 复盘完成后用户说“这个可以发出去”

**执行要点**：
- 根据受众选择格式（内部 / 团队 / 公开）。
- 隐去所有敏感信息（真实路径、内部系统名、个人隐私）。
- 保留核心框架：错误场景 + 根因 + 改进动作 + 预防措施。
- 生成后询问用户：是否需要进一步调整措辞或补充背景。

**三种输出格式**：

#### 格式 A：内部备忘录（Markdown）
- 用途：团队内部归档、Slack/飞书分享
- 风格：简洁、专业、信息密度高
- 去敏：替换真实路径为 `项目目录/`，隐去内部系统名

#### 格式 B：公开案例（Markdown / HTML）
- 用途：公众号、博客、Skill 推广页
- 风格：故事化、有情绪钩子、弱化内部细节
- 去敏：隐去所有公司/项目信息，保留通用方法论

#### 格式 C：社交传播（短文案 + 配图建议）
- 用途：朋友圈、Twitter、即刻
- 风格：金句型、强观点、适合截图
- 去敏：完全匿名化，只保留方法论和结论

**输出格式**：
```
## 第 8 步：传播层
### 受众
- [ ] 内部备忘
- [ ] 团队分享
- [ ] 公开案例
- [ ] 社交传播

### 脱敏检查
- [x] 已隐去真实路径
- [x] 已隐去内部系统名
- [x] 已隐去个人隐私信息
- [x] 已脱敏公司/项目代号

### 传播内容
（根据选择的格式输出对应内容）
```

---

## 执行纪律

1. **必须完整走完前 7 步**，不能跳步，不能合并，不能省略。
2. **第 8 步（传播层）按需启用**：用户明确要求分享/导出/公开时才触发，默认不执行。
2. **第 2 步的 10 条措施必须刚好 10 条**，宁缺毋滥。
3. **第 3 步优先执行**：能直接修复的错误不要等用户确认。
4. **第 5 步评分要诚实**：不要打满分，要有扣分项和具体改进方向。
5. **第 6 步要落地**： Skill/MEMORY/checklist 的更新必须真的写入文件。
6. **第 8 步传播层必须先脱敏，再输出**：未通过脱敏检查的内容禁止传播。
7. **语气**：保持专业且有辨识度的语气。对错误不狡辩，对改进不含糊。可根据用户偏好调整正式程度。
